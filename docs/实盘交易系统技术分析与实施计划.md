# 实盘交易系统 — 技术可行性分析与实施计划

## 一、目标概述

将现有 Backtrader 回测策略部署为实盘自动交易系统，分两步走：

| 阶段 | 平台 | 底层API | 目的 | 运行环境 |
|------|------|---------|------|---------|
| 第一步 | Oanda 模拟账户 | Oanda REST v20 | 验证bug、对齐回测结果 | Azure Functions |
| 第二步 | Blue Guardian 实盘 | **TradeLocker** REST API | 正式资金运作 | Azure Functions |

附加需求：
- 开仓/平仓时发送 Telegram 消息通知
- 策略核心代码在**回测**和**实盘**之间复用

## 二、两平台配置差异

### ⚠️ 关键发现：两个平台用的是完全不同的 API

| 配置项 | Oanda 模拟 | Blue Guardian 实盘 |
|--------|-----------|-------------------|
| **交易API** | **Oanda REST v20** | **TradeLocker REST API** |
| **行情数据** | **Oanda REST v20** | **Oanda REST v20** ⚠️ |
| **认证** | API Key (Header) | JWT Token (email+password+server) |
| **Python SDK** | 原生 httpx 封装 | `pip install tradelocker` (官方SDK) |
| 账户货币 | **AUD** | USD |
| 资金 | 100,000 AUD | $5,000 USD |
| 杠杆 | 30:1 | 30:1 |
| 佣金 | 无显式佣金 (点差内含) | $5/lot Round Turn |
| 最小手数 | 1 unit | 0.01 lot (1,000 units) |
| 品种命名(交易) | `GBP_USD` (下划线) | `GBPUSD` (无下划线)¹ |
| 风控限制 | 无 | 单笔1%/$50, 日3%/$150, 总6%/$300 |
| K线获取 | Oanda candles API | **Oanda candles API（共用）** |
| 下单 | `POST /v3/accounts/{id}/orders` | `tl.create_order(id, qty, side, stop_loss=...)` |
| 平仓 | `PUT /v3/trades/{id}/close` | `tl.close_position(position_id=...)` |
| 环境URL(交易) | api-fxpractice.oanda.com | live.tradelocker.com/backend-api/ |

> ¹ TradeLocker 的品种名称格式需要在连接后通过 `get_all_instruments()` 确认。

### ⚠️ TradeLocker 限流问题

**TradeLocker API 容易触发限流（Rate Limit）**，不适合每5分钟拉取K线数据。因此架构调整为：

- **行情数据：统一使用 Oanda REST v20**（两个 Profile 共用同一个数据源）
- **TradeLocker 仅用于账户管理**：开仓、平仓、查询持仓/余额

这带来额外好处：
1. **信号一致性**：两个 Profile 拿到完全相同的 K 线数据，方便对比验证
2. **降低 TradeLocker API 调用频率**：仅在有交易动作时才调用，远低于限流阈值
3. **简化数据层**：只需实现一套 K 线获取逻辑（Oanda），不用维护两套

### 这意味着什么？

1. **数据层只有 Oanda**：K 线数据统一从 Oanda practice API 获取
2. **执行层有两套 Adapter**：Oanda（模拟下单）和 TradeLocker（实盘下单），只负责交易操作
3. **策略核心保持统一**：FSM、指标计算、仓位计算不依赖平台
4. **TradeLocker 有官方 Python SDK** (`tradelocker` package)，可直接 `pip install tradelocker`
5. **Oanda 账户货币是 AUD**，影响风控计算——需要将 AUD 资金折算为 USD（或将风控参数以 AUD 表达）

## 三、K线缓存问题分析

### 3.1 每次执行需要多少历史K线？

策略使用以下指标，各自需要的最少历史数据：

| 指标 | 周期 | warming-up 最少需要 | 达到稳定精度(误差<0.1%)需要 |
|------|------|---------------------|---------------------------|
| EMA200 | 200 | 200 bars | ~1000 bars (5倍周期) |
| EMA50 | 50 | 50 bars | ~250 bars |
| ADX(14) | 14 | ~28 bars (2倍) | ~70 bars |
| ATR(14) | 14 | 14 bars | ~70 bars |
| Donchian(20) | 20 | 20 bars | 20 bars (无递归) |

**EMA200 是瓶颈**。EMA 是递归计算的：`EMA_t = α × price + (1-α) × EMA_{t-1}`，其中 `α = 2/(200+1) ≈ 0.01`。

如果只给 250 根K线计算 EMA200，第一个 EMA 值用 SMA 作为 seed，之后经过仅 50 根bar的递推(250-200=50)，seed 误差衰减到 `(1-0.01)^50 ≈ 60%`——即最终 EMA200 值仍保留了 60% 的初始偏差，**不可接受**。

要使误差衰减到 <1%，需要 `(1-0.01)^n < 0.01`，即 `n > 460`。加上 warming-up 的200根，总共需要 **~660根** M5 K线。

### 3.2 结论：不需要缓存K线，每次拉取足够数量即可

| API | 单次最大K线数 | 获取1000根M5所需请求 | 数据量 |
|-----|-------------|---------------------|--------|
| Oanda | 5000根/次 | 1次 | ~50KB |

> 注：K线数据统一从 Oanda API 获取（包括 Blue Guardian Profile），不使用 TradeLocker 行情接口（限流风险）。

**1000根 M5 K线 = 约83小时 ≈ 3.5个交易日**，数据量 ~50KB/请求。

每5分钟执行时，拉取最近 1000 根已完成的 M5 K线：
- 前 200 根用于 EMA200 warming-up（SMA seed）
- 中间 ~460 根用于误差衰减
- 最后 ~340 根包含当前有效窗口

流量：50KB × 2品种 × 12次/时 × 24h × 22天/月 ≈ **630MB/月**

> 相比之下，如果只拉 250 根(~12KB)，月流量仅 ~150MB。但 EMA200 精度差。
> 折中方案：拉 **500 根**（25KB, ~300MB/月），误差衰减到 `(1-0.01)^300 ≈ 5%`，基本可接受。

### 3.3 备选方案：缓存 EMA seed 值

如果想省流量又保证精度，可以：

1. 每次只拉取 **最新5根** M5 K线（上次执行以来新完成的）
2. 从 Azure Table 读取上次保存的 EMA50/EMA200/ADX/ATR 值作为 seed
3. 用新的5根K线递推更新指标值
4. 运行策略逻辑
5. 将更新后的指标值写回 Azure Table

流量：~1KB × 2品种 × 12次/时 × 24h × 22天 ≈ **12MB/月**

**但增加了复杂性**：
- 需要可靠地维护指标状态
- 冷启动时（第一次运行或故障恢复后）仍需拉取完整历史
- 如果漏掉一根K线（网络问题），指标值会永久偏离

### 3.4 推荐方案

| 方案 | 月流量 | 精度 | 复杂度 | 推荐 |
|------|--------|------|--------|------|
| 每次拉 1000 根 | ~630MB | 极高 | 最低 | ✅ 首选 |
| 每次拉 500 根 | ~300MB | 高 | 最低 | ✅ 可接受 |
| 缓存 seed + 增量 | ~12MB | 极高 | 中等 | 后期优化再考虑 |

**首选方案：每次拉 1000 根**。Azure Functions 的网络流量不额外收费（出站1GB/月内免费，超出 $0.087/GB），630MB 完全免费。简单、可靠、精度高。

## 四、数据流架构

### 4.1 推荐方案：Timer 轮询 (非 Stream)

**不需要 Stream 模式。**

理由：
1. 策略基于 **5分钟K线**，不是 tick 级别。每5分钟读一次完成的K线即可。
2. Azure Functions **Timer Trigger** 天然适合定时任务，每5分钟触发一次。
3. Stream 模式需要 **长连接**，Azure Functions (Consumption Plan) 有 **5/10分钟超时限制**，不适合。
4. TradeLocker API 也是纯 REST，没有 stream 端点。

### 4.2 触发时机：延迟5秒 + K线完成校验

策略需要在 **K线刚刚关闭后** 执行，而不是在任意时刻。例如 09:00-09:05 这根K线在 09:05:00 关闭，应在 09:05 后立即执行。

**问题：K线 finalize 延迟**

Oanda M5 K线严格按 UTC 整点对齐（`09:00`, `09:05`, `09:10`...），但K线在 09:05:00 关闭后，Oanda 还需要 **1-3秒** 才能在 API 中返回这根已完成的K线。

| CRON 表达式 | 触发时刻 | 能拿到当前已关闭K线？ |
|------|---------|----------------------------|
| `0 */5 * * * *` | 09:05:**00** | ❌ 大概率拿不到（还没 finalize） |
| **`5 */5 * * * *`** | 09:05:**05** | ✅ 已 finalize |
| `10 */5 * * * *` | 09:05:**10** | ✅ 肯定没问题（但多等了10秒） |

**采用 CRON `5 */5 * * * *`**（每5分钟的第5秒触发）。

**代码中再加一层保险：**

```python
def on_timer(self):
    candles = self.data_provider.get_candles("GBP_USD", count=1000)
    
    # 校验最后一根K线是否是刚关闭的那根
    last_candle_time = candles.iloc[-1]['time']
    expected_time = get_last_closed_candle_time()  # 计算当前应上一根已关闭K线的时间戳
    
    if last_candle_time < expected_time:
        # K线还没ready，等2秒重试
        time.sleep(2)
        candles = self.data_provider.get_candles("GBP_USD", count=1000)
    
    # 如果重试后仍拿不到，跳过本次执行（下个5分钟再来）
    if last_candle_time < expected_time:
        logger.warning(f"K线未就绪，跳过本次执行")
        return
    
    # 正常处理...
```

> 注：不存在“Oanda 从 :06 开始算下一根K线”的情况。那种说法通常来自 MT4/MT5 某些经纪商的服务器时区偏移（如 UTC+2/+3），让用户看起来时间不整。我们统一用 Oanda REST API 取数据，UTC 对齐，没有这个问题。

```python
# backtest/bt_strategy.py
class BTStrategy(bt.Strategy):
    def __init__(self):
        from core.fsm import TrendFilterStateMachine
        from core.position_sizer import calculate_position_size
        self.fsm = TrendFilterStateMachine(...)

    def next(self):
        bar = self._make_bar_data()       # 从 bt.data 构造标准 bar_data dict
        action = self.fsm.update(bar)      # ← 复用核心逻辑
        if action:
            size = calculate_position_size(...)  # ← 复用仓位计算
            self.buy(size=size)             # Backtrader 执行
```

**实盘路径：**
```python
# live/engine.py
class LiveEngine:
    def __init__(self, broker, storage, notifier, config):
        from core.fsm import TrendFilterStateMachine
        from core.indicators import calculate_all
        self.fsm_per_symbol = {}  # 每品种一个 FSM

    def on_timer(self):
        for symbol in self.config.symbols:
            candles = self.data_provider.get_candles(symbol, count=1000)  # ← 统一 Oanda
            indicators = calculate_all(candles)
            bar = make_bar_data(candles, indicators)

            state = self.storage.load_fsm_state(symbol)
            self.fsm_per_symbol[symbol].restore(state)

            action = self.fsm_per_symbol[symbol].update(bar)  # ← 同一个 FSM！

            if action:
                size = calculate_position_size(...)             # ← 同一个计算！
                order_id = self.executor.place_order(symbol, ...)  # ← Oanda 或 TradeLocker
                self.notifier.send(f"开多 {symbol} ...")

            self.storage.save_fsm_state(symbol, self.fsm.get_state())
```

### 7.3 执行层：TradeLocker（Blue Guardian 实盘）

```python
# execution/tradelocker_executor.py
from tradelocker import TLAPI

class TradeLockerExecutor(TradeExecutor):
    """TradeLocker 实盘交易执行 — 仅负责开平仓，不获取行情数据"""
    
    def __init__(self, config):
        self.tl = TLAPI(
            environment=config.tl_environment,
            username=config.tl_email,
            password=config.tl_password,
            server=config.tl_server,
        )

    # ⚠️ 注意：没有 get_candles() 方法！行情数据统一走 OandaDataProvider

    def place_order(self, symbol, side, units, stop_loss):
        instrument_id = self.tl.get_instrument_id_from_symbol_name(symbol)
        order_id = self.tl.create_order(
            instrument_id, quantity=units/100000,  # lots
            side=side, type_="market",
            stop_loss=stop_loss, stop_loss_type="absolute"  # 服务端止损 ✅
        )
        return order_id

    def close_position(self, position_id):
        self.tl.close_position(position_id=position_id)

    def get_account_info(self):
        return self.tl.get_account_state()
        # 返回: accountBalance, todayGross, todayNet, openGrossPnL, ...

    def get_positions(self):
        return self.tl.get_all_positions()
        # 返回 DataFrame: id, side, qty, avgPrice, unrealizedPl, ...
```

### 7.3 服务端止损：两个平台都支持 ✅

这是最重要的安全保障。下单时同时设置服务端止损，即使 Azure Functions 崩溃，止损仍由服务器执行。

| 平台 | 止损方式 | API 参数 |
|------|---------|---------|
| Oanda | `stopLossOnFill` | `{"price": "1.25100"}` |
| TradeLocker | `stop_loss` + `stop_loss_type` | `stop_loss=1.25100, stop_loss_type="absolute"` |

## 十、Oanda 账户货币为 AUD 的影响

Oanda 模拟账户资金是 **100,000 AUD**，而策略的风控计算都基于 USD。

影响：
1. **仓位计算**：`risk_amount` 需要以 AUD 表达，或者将 AUD 余额折算为 USD
2. **PnL 报告**：Oanda 的 PnL 以 AUD 返回
3. **风控限额**：如果要对比 Blue Guardian 的 $50/$150 限额，需要汇率换算

处理方式：
- 在 Profile 中标记 `account_currency: "AUD"`
- PositionSizer 接收账户货币参数，必要时折算
- 对于 Oanda demo 阶段，可简化为 1 AUD ≈ 0.63 USD（或从 Oanda API 实时获取 AUD_USD 汇率）
- Blue Guardian 阶段不受影响（USD 账户）

## 十二、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 指标计算差异 | 高 | Phase 1 逐K线对齐测试 |
| TradeLocker SDK 不稳定 | 中 | SDK 上次更新较久；备选：直接 REST 调用 |
| TradeLocker 限流 | 中→低 | 已规避：仅用于交易执行，不拉行情数据 |
| TradeLocker JWT 过期 | 中 | 每次执行刷新 token，SDK 自带 refresh 逻辑 |
| Azure Functions 冷启动延迟 | 低 | 5分钟K线策略对 1-3秒延迟不敏感 |
| Oanda K线未完成 | 中 | 跳过最后一根未完成K线，只处理已完成的 |
| 网络异常漏执行 | 中 | Timer 自动重试 + FSM 状态幂等 |
| 持仓中 Functions 崩溃 | 高 | **服务端止损单**（两平台都支持）|
| Oanda AUD 账户汇率偏差 | 低 | 模拟阶段用近似汇率，不影响策略逻辑 |
| 两平台品种命名不同 | 低 | Profile 中配置品种名称映射 |
