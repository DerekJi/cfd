# 半自动交易策略逻辑 V3.0

## 一、核心架构：双轨运行模式

### 慢轨 (Slow Track - 5min/15min 周期)

- **任务**：计算技术指标（ATR、EMA）及自动绘线辅助
- **人工干预**：
  - 根据大级别趋势，划定 UP（压力线）和 DOWN（支撑线）
  - **人工判断**当前市场模式：震荡模式 / 趋势模式
  - **方向限制**：标记是否允许做多/做空（防止与高级别周期冲突）
    - 例如：日线下跌趋势中，禁止5分钟做多

### 快轨 (Fast Track - Oanda Stream Tick 级)

- **任务**：运行状态机，实时监控价格相对于 UP/DOWN 的位置及其维持时间

---

## 二、震荡模式 (Range Mode)：五步状态机

为了规避 5 分钟 K 线内产生的瞬时"毛刺"，我们将入场逻辑拆解为以下状态转移：

### 状态 0: IDLE (静默)

- **监测**：当前价格
- **触发条件**：若 `Price > UP`，记录当前时间戳并转入状态 1

### 状态 1: TOUCHING (触碰中)

- **维持判定**：若价格持续在 UP 之上超过 N 秒（建议 30 秒），转入状态 2
- **重置判定**：若价格跌回 UP 之下，说明是瞬时干扰，重置回状态 0

### 状态 2: TOUCHED_SUCCESS (有效突破)

- **标记**：步骤 A 完成
- **记录**：突破后的最高点 `Peak_Price`
- **转移条件**：当价格回落至 `UP - Buffer`（Buffer 设为 $0.2 \times ATR$）时，转入状态 3

### 状态 3: RETRACED (回落确认)

- **记录**：回落形成的局部低点 `Dip_Price`
- **入场触发**：价格反转向上，一旦突破 `Peak_Price`（或前一根 5min K 线高点），立即执行买入
- **失效判定**（满足任一条件则取消交易，回到状态 0）：
  - 价格跌破区间中轴：`Dip_Price < (UP + DOWN) / 2`
  - **假突破防御**：价格重新跌破UP线：`Price < UP`
  - 回落时间过长：超过30分钟未触发入场

### 状态 4: POSITION (持仓管理)

- **初始止损**：设置于 `Dip_Price - Spread`
- **移动止损 (Trailing Stop)**：
  - **固定止损额度** = `开仓价 - Dip_Price`
  - **更新频率**：每根5分钟K线收盘时检查
  - **移动规则**：若产生新高，则止损移至 `新高 - 固定止损额度`
  - 注意：止损只能向上移动，不能向下调整

## 三、趋势模式 (Trend Mode)：动态修正

当行情被标记为趋势时，程序切换为"均线回踩"逻辑：

### 趋势过滤器

- **条件**：$EMA_{20} > EMA_{60}$ 且 $EMA_{20}$ 斜率向上

### 入场点

- **触发**：价格回落触碰 $EMA_{20}$

### 触发动作

- **策略**：不直接买入，需满足以下确认信号之一：
  - **K线形态**：5分钟级别出现阳线反弹（收盘价高于前一根K线中值）
  - **Pin Bar确认**：触碰EMA20后，下一根K线形成多头Pin Bar（下影线 > 实体的2倍）
  - **动量过滤**：RSI从超卖区（<40）反弹至45以上

### 止损设置

- **位置**：设在 $EMA_{60}$ 附近或前一个震荡区间的边缘

## 四、风险管理与全局约束

### 仓位自动计算

- **计算逻辑**：程序根据账户净值（Equity）和止损距离（`Entry - Dip_Price`），自动换算成 Oanda 的 Units
- **风险控制**：确保单笔风险控制在 1% 左右

### 时段限制

- **激活时段**：14:30 - 23:00 (ACDT)
- **原因**：考虑到阿德莱德的时间安排，避开清晨流动性枯竭阶段

### 消息路由

- **状态 1 → 2**：发送 Telegram "关注提醒"
- **状态 3 → 4**：发送 Telegram "已入场，请确认"

---

## 五、全局交易规则

### 方向限制检查（每次状态转换前必检）

- **做多前提**：
  - 高级别（日线/4小时）趋势向上，或处于震荡区间上半部分
  - 人工标记允许做多
- **做空前提**：
  - 高级别（日线/4小时）趋势向下，或处于震荡区间下半部分
  - 人工标记允许做空
- **禁止双向**：当高级别趋势明确时，只允许顺势方向交易

---

## 六、针对 Oanda API 的特别调整

### 移除"放量"条件

- **原因**：鉴于 CFD 成交量的局限性
- **替代方案**：将确认条件锁定在价格动量（突破 `Peak_Price` 的速度）上

### Buffer 设置

- **策略**：利用 ATR 动态调整 Buffer
- **目的**：避免在低波动期（亚盘）因为几个点的波动频繁切换状态
