# 平仓逻辑修复说明

**日期**: 2026-02-18  
**版本**: v1.0.3  
**问题**: 实盘运行中大量平仓失败消息

---

## 问题根本原因

### 1. 平仓逻辑未验证持仓存在性
回测代码改为实盘时，平仓逻辑直接继承了回测的做法：
- FSM 触发平仓信号 → 直接调用 `executor.close_position(trade_id)`
- **未先查询 Oanda 确认持仓是否存在**

实盘环境中：
- 止损/止盈由平台服务端自动执行（开仓时已挂单）
- 平台自动平仓后，`trade_id` 已失效
- 代码的 FSM 仍处于 `POSITION_LONG/SHORT` 状态
- 下次 tick 继续触发平仓信号 → **重复平仓已不存在的持仓** → 失败

### 2. FSM 状态与实际持仓不同步
- 平台自动平仓时，代码无感知
- Storage 中的持仓记录未清理
- FSM 状态未重置为 IDLE
- 导致：
  - 无法开新仓（FSM 仍在 POSITION 状态）
  - 重复尝试平仓（每次 tick 都触发平仓逻辑）

### 3. 硬止损检查冗余
实盘中，止损已在开仓时挂服务端订单：
```python
'stopLossOnFill': {'price': price_str}
```
代码中保留回测的硬止损检查逻辑是多余的，且可能导致：
- 尝试平仓已被平台平掉的持仓
- 产生不必要的 API 调用和错误日志

---

## 修复方案

### 1. 每次 tick 开始时同步持仓状态

**文件**: `live/engine.py` - `tick()` 方法

在处理品种前，增加持仓同步：
```python
# 【核心修复】同步持仓状态：清理已平仓的记录和FSM状态
self._sync_positions()
```

**新增方法**: `_sync_positions()`
- 调用 `executor.get_positions()` 获取实盘持仓
- 对比每个品种的 Storage 记录
- 如果 Storage 有记录但实盘无持仓 → **已被平台自动平仓**
  - 删除 Storage 持仓记录
  - 重置 FSM 状态为 IDLE
  - 记录日志
- 如果实盘有持仓但 Storage 无记录 → **异常情况（手动开仓？）**
  - 记录错误日志
  - 发送告警通知

### 2. 平仓前验证持仓存在

**文件**: `live/engine.py` - `_handle_exit()` 方法

修改逻辑：
```python
# 【核心修复】先验证持仓是否真实存在
if not self.config.dry_run:
    # 检查持仓是否仍然存在
    actual_positions = self.executor.get_positions()
    position_exists = any(p.position_id == trade_id for p in actual_positions)
    
    if not position_exists:
        # 已被自动平仓，只清理状态，不执行平仓
        logger.warning("持仓已不存在（可能已被止损/止盈自动平仓），仅清理状态")
        # 使用当前价作为退出价记录
        exit_price = bar_data['close']
        action['reason'] = f"{action.get('reason')}_auto_closed"
    else:
        # 持仓存在，执行平仓
        close_result = self.executor.close_position(trade_id)
```

**关键点**：
- 不存在的持仓不尝试平仓（避免重复失败）
- 仍记录交易结果（用于统计和风控）
- 标记平仓原因为 `auto_closed`

### 3. 移除硬止损检查

**文件**: `live/engine.py` - `_process_symbol()` 方法

移除回测遗留代码：
```python
# 【已移除硬止损检查】实盘完全依赖服务端止损，不需要代码再次平仓
#    平台自动平仓后，_sync_positions() 会自动清理状态
```

**理由**：
- 实盘止损由服务端挂单处理
- 避免冗余的平仓尝试和错误日志
- 简化代码逻辑

### 4. 处理 FSM 状态不一致

**文件**: `live/engine.py` - `_handle_exit()` 方法

增加状态一致性检查：
```python
if not pos:
    logger.warning(f"{symbol}: 收到平仓信号但无持仓记录")
    # FSM状态需要重置为IDLE
    if fsm.state.name.startswith('POSITION'):
        fsm._reset()
        self._save_fsm_state(symbol, fsm)
        logger.info(f"{symbol}: FSM状态已重置为IDLE")
    return {'action': 'no_position'}
```

## 新增工具：状态修复脚本

**文件**: `live/fix_state.py`

用途：手动修复 Storage 中的 FSM 状态和持仓记录

```bash
# 检查状态（只读）
python -m live.fix_state --profile oanda_demo --dry-run

# 执行修复
python -m live.fix_state --profile oanda_demo
```

功能：
1. 获取实盘持仓
2. 对比 Storage 记录
3. 清理不一致的记录
4. 重置 FSM 状态
5. 生成修复报告

使用场景：
- 故障恢复后的状态一致性检查
- 手动开仓/平仓后的状态同步
- 定期维护和校验

   - `_handle_exit()`: 增加持仓存在性验证和状态一致性处理
