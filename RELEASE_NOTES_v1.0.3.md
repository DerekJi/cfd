# Release Notes v1.0.3

**发布日期**: 2026-02-18  
**类型**: 🔥 紧急修复 (Hotfix)

---

## 概述

修复实盘运行中的平仓失败问题。主要原因是回测逻辑直接迁移到实盘，未考虑平台自动止损/止盈的特性，导致代码重复尝试平仓已被平台自动平掉的持仓。

**影响范围**: 所有实盘环境  
**严重程度**: 高（影响正常交易和状态管理）  
**升级建议**: ⚠️ **必须立即升级并运行状态修复工具**

---

## 🐛 Bug 修复

### 1. 平仓逻辑验证持仓存在性
**问题**: FSM 触发平仓信号后，直接调用 `close_position()`，未验证持仓是否真实存在

**原因**:
- 实盘中，止损/止盈由平台服务端自动执行
- 平台自动平仓后，`trade_id` 已失效
- 代码的 FSM 仍处于持仓状态，继续尝试平仓

**修复**: `live/engine.py` - `_handle_exit()`
- 平仓前先查询实盘持仓列表
- 如持仓不存在 → 标记为自动平仓，只清理状态，不执行平仓
- 如持仓存在 → 正常执行平仓

**影响**: 消除大量 "平仓失败" 错误消息

---

### 2. 增加持仓状态同步机制
**问题**: 平台自动止损/止盈后，Storage 和 FSM 状态未更新

**原因**:
- 代码无法感知平台的自动平仓
- FSM 仍处于 `POSITION_LONG/SHORT` 状态
- 导致无法开新仓或重复平仓

**修复**: `live/engine.py` - 新增 `_sync_positions()`
- 每次 tick 开始时，对比实盘持仓与 Storage 记录
- 清理已平仓的记录
- 重置对应品种的 FSM 状态为 IDLE
- 记录异常情况（如实盘有持仓但 Storage 无记录）

**影响**: FSM 状态与实盘持仓保持同步，可正常开新仓

---

### 3. 移除实盘硬止损检查
**问题**: 保留了回测的硬止损检查逻辑，在实盘中是冗余的

**原因**:
- 实盘开仓时已挂服务端止损订单
- 不需要代码再次检查和平仓

**修复**: `live/engine.py` - `_process_symbol()`
- 移除 `check_hard_stop()` 调用
- 简化代码逻辑

**影响**: 减少冗余 API 调用，避免重复平仓尝试

---

### 4. 增强状态一致性处理
**问题**: FSM 触发平仓信号但 Storage 无持仓记录时，FSM 状态未重置

**修复**: `live/engine.py` - `_handle_exit()`
- 检查 FSM 是否处于持仓状态
- 如是，则重置为 IDLE 并保存

**影响**: 防止 FSM 状态卡住

---

## ✨ 新功能

### 状态修复工具
**文件**: `live/fix_state.py`

提供手动修复 Storage 和 FSM 状态的工具：

```bash
# 检查状态（只读）
python -m live.fix_state --profile oanda_demo --dry-run

# 执行修复
python -m live.fix_state --profile oanda_demo
```

**功能**:
- 对比实盘持仓与 Storage 记录
- 清理不一致的持仓记录
- 重置 FSM 状态为 IDLE
- 生成详细修复报告

**使用场景**:
- 故障恢复后的状态一致性检查
- 手动开仓/平仓后的状态同步
- 定期维护和校验
